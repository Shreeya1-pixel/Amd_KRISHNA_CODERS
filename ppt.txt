SLIDE 3 — THE SAFEO ANSWER (ONE SLIDE PITCH)
================================================================================

TITLE:
  Inline decision engine — not a chatbot wrapper

BULLETS:
  • Real-time scan at point of entry (Odoo, REST API, website, WhatsApp-style)
  • Three outcomes: ALLOW · WARN · BLOCK

  • Self-improving: Bayesian thresholds + LoRA controller (ROCm bf16)
  • Zero OpenAI dependency — Fireworks on AMD hardware for agent reasoning

BIG STAT ROW:
  5 agents · 3 tiers · 15 attack-graph nodes · 0 OpenAI keys

        └──────────────────┬───────────────────┘
                           │ still ambiguous?
                           ▼
        ┌──────────────────────────────────────┐

BULLETS:
  • Most scans resolve at Tier 1 — fast, cheap, deterministic
  • Tier 2 escalates only when needed — saves tokens & latency
  • Tier 3 for semantic edge cases — Fireworks Llama / DeepSeek-R1

APPENDIX B — ANTICIPATED JUDGE QUESTIONS & ANSWERS
================================================================================

Q: How is this different from a ChatGPT wrapper?
A: Every Assistant message hits POST /v1/scan — deterministic tiers first, agents
   only on WARN/BLOCK. UI adapts; engine is fixed forensic pipeline with audit hash.

Q: Where is AMD used if you didn't get cloud credits?
A: Fireworks runs on AMD Instinct GPUs — our production LLM path. Codebase is
   ROCm-ready for local Tier 2 + LoRA; amd_setup/ activates when pod is provisioned.

Q: False positives?
A: VerifierAgent downgrades BLOCK→WARN on disagreement; Bayesian engine learns from
   analyst feedback; thresholds clamped [0.45, 0.90].

Q: Containerized?
A: Yes — docker-compose.yml, backend Dockerfile, frontend nginx Dockerfile, Render deploy.

Q: Gemma / Fireworks credits?
A: Fireworks integration for agent LLM + optional Gemma path via same API; tiered
   design minimizes token spend.

Q: Real ERP integration?
A: Odoo module (securec_odoo) proxies to SafeO API; dashboard Connect flow; website
   is standalone demo of same engine.
